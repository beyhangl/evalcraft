    Twitter/X Launch Threads¶

    Thread 1: Problem-focused — "Your AI agents are untested"¶

    Tweet 1 Your AI agents are untested.

    Not "undertested." Untested.

    You ship features, you vibe-check them in a Jupyter notebook, and you call it done.

    Here's why that's a problem — and what we built to fix it. 🧵

    Tweet 2 Running 200 tests against GPT-4 costs real money. Every. Single. Commit.

    At $0.0008/run, a modest test suite costs $5+ per push. CI/CD gates on agent evals? Economically indefensible.

    So you don't test. And your agents regress silently.

    Tweet 3 Even if you pay, LLMs aren't deterministic.

    The same test passes on Monday and fails on Friday. Not because your code changed — because the model drifted.

    You can't trust a test that flips randomly. So you ignore it. Which means you're not testing at all.

    Tweet 4 The root problem: agent testing tools were built for observability, not for CI.

    Braintrust, LangSmith — great for production tracing. Terrible for fast, free, deterministic test suites.

    Nobody built the pytest equivalent. Until now.

    Tweet 5 Introducing evalcraft.

    Capture your agent run once as a cassette (JSON file, like VCR for HTTP). Replay it deterministically — zero API calls, zero cost.

    Your 10-minute, $5 test suite becomes 200ms and $0.

    pip install "evalcraft[pytest]"

    Tweet 6 What a cassette looks like in a pytest test:

    def test_weather_agent():
    run = replay("tests/cassettes/weather.json")
    assert assert_tool_called(run, "get_weather").passed
    assert assert_cost_under(run, max_usd=0.01).passed

    Run with pytest. It's just Python. 200ms. $0.00. Every time.

    Tweet 7 Cassettes are plain JSON. Check them into git. Diff them in PRs. The evalcraft diff CLI shows exactly what changed between runs:

    Tool sequence: ['search'] → ['search', 'send_alert']
Output text changed
Tokens: 135 → 210

    Regression detection, for free, in your existing workflow.

    Tweet 8 Open source. MIT. No SaaS. No dashboard you'll forget to check.

    Just a Python package that makes agent testing as normal as pytest tests/.

    GitHub: https://github.com/beyhangl/evalcraft pip install evalcraft

    Thread 2: Technical deep-dive — "How we built capture-replay for agents"¶

    Tweet 1 VCR.py changed HTTP testing in 2012.

    Record once, replay forever. No real network. Deterministic CI.

    We built the same thing for AI agents. Here's how it works technically. 🧵

    Tweet 2 The core abstraction is a Cassette — a JSON document that captures the full trace of an agent run:

    Cassette
├── input_text, output_text
├── total_cost_usd, total_tokens
├── fingerprint (SHA-256)
└── spans[]
    ├── llm_request/llm_response
    └── tool_call (name, args, result, duration_ms)

    Every LLM call and tool invocation, recorded.

    Tweet 3 The CaptureContext API is explicit by design.

    You wrap your agent run, record each event, and the cassette serializes to JSON:

    with CaptureContext(name="run", save_path="cassettes/run.json") as ctx:
    ctx.record_input("What's the weather in Paris?")
    ctx.record_tool_call("get_weather", args={"city": "Paris"}, result={"temp": 18})
    ctx.record_llm_call(model="gpt-4o", input="...", output="...", cost_usd=0.0008)
    ctx.record_output("It's 18°C in Paris.")

    Explicit > implicit. You know exactly what gets recorded.

    Tweet 4 Framework adapters patch at the SDK level so you don't have to instrument manually.

    For OpenAI: 

    from evalcraft.adapters.openai import patch_openai
patch_openai(openai)  # All subsequent calls auto-recorded

    For LangGraph: 

    handler = EvalcraftCallbackHandler()
graph.invoke(input, config={"callbacks": [handler]})

    One line. Works with existing code.

    Tweet 5 Replay is zero-network by definition — it reads spans from the cassette and reconstructs the run state.

    The fingerprint field (SHA-256 of span content) lets you detect if the agent's behavior changed between captures. Regression detection is just fingerprint comparison.

    Tweet 6 The assertion layer is composable:

    assert assert_tool_called(run, "get_weather", with_args={"city": "Paris"}).passed
assert assert_tool_order(run, ["search", "summarize"]).passed
assert assert_cost_under(run, max_usd=0.05).passed

    Each returns a result object with .passed and .message — pytest-native, no magic.

    Tweet 7 Mock LLM supports exact match, glob patterns, and wildcards so you can test agent logic without a live model:

    llm = MockLLM()
llm.add_response("*weather*", "It's sunny and 22°C.")
llm.add_response("*", "I don't know.")  # fallback

    Test your agent's decision logic, not the LLM's output.

    Tweet 8 The result: a pytest test suite for agents that runs in ~200ms and costs $0.

    If you're building agents and not testing them with evalcraft (or something like it), you're shipping blind.

    https://github.com/beyhangl/evalcraft

    Thread 3: Comparison — "I tested 5 agent eval tools. Here's what's missing."¶

    Tweet 1 I spent 2 weeks evaluating every major agent eval tool.

    LangSmith, Braintrust, Promptfoo, DeepEval, custom scripts.

    Here's what none of them get right — and what evalcraft does differently. 🧵

    Tweet 2 First, the landscape:

    • LangSmith — excellent observability, production tracing. Built for monitoring.
    • Braintrust — great human eval UX, experiment tracking. Built for evals-as-experiments.
    • Promptfoo — YAML-configured, good for prompt regression. Not agent-aware.
    • DeepEval — LLM-as-judge focus. Requires live API calls per test run.

    All solid tools. None of them solve CI.

    Tweet 3 The CI problem:

    None of these tools let you run a deterministic, zero-cost test suite on every commit.

    They all require live API calls to "eval." That means: slow, expensive, non-deterministic.

    You can't gate a deploy on a test that costs $5 and takes 10 minutes.

    Tweet 4 The real gap: agent testing ≠ agent observability.

    Observability = "what happened in production?" Testing = "did my code change break the agent?"

    These are different problems. The first needs a dashboard. The second needs pytest.

    No tool treated them as different — until evalcraft.

    Tweet 5 Evalcraft's answer: cassette-based replay.

    Record the agent run once (including all LLM + tool calls). Save it as JSON. Replay it deterministically forever.

    pytest tests/  # 200ms, $0.00, deterministic

    That's CI-friendly agent testing.

    Tweet 6 The comparison table no one published:

    Evalcraft Braintrust LangSmith Promptfoo
    Cassette replay ✅ ❌ ❌ ❌
    Zero-cost CI ✅ ❌ ❌ Partial
    pytest-native ✅ ❌ ❌ ❌
    Mock LLM/Tools ✅ ❌ ❌ ❌
    Observability ❌ ✅ ✅ ❌

    Use LangSmith/Braintrust for production tracing. Use evalcraft for CI.

    Tweet 7 The mock system is what makes it practical.

    MockLLM + MockTool = test your agent's routing logic, error handling, and tool sequencing without touching a real API.

    If you've ever written unittest.mock.patch for an LLM call... you know this pain.

    Tweet 8 Bottom line: agent tooling has been obsessed with evaluation UX and dashboards.

    Nobody built the unsexy thing: a fast, free, deterministic test runner you can run in CI.

    evalcraft is that tool. MIT. Open source. pip install evalcraft.

    https://github.com/beyhangl/evalcraft
