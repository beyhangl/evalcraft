    Product Requirements Document (PRD)¶

    Evalcraft — The Agent Testing & Evaluation Platform¶

    Task #4 | Author: product-lead | Date: March 2026

    1. Executive Summary¶

    Evalcraft is the "pytest for AI agents" — an open-source testing framework and cloud platform that lets developers capture, replay, mock, and evaluate AI agent behavior with deterministic, sub-cent assertions that run in CI/CD on every commit.

    The market has $400M+ invested across 20+ competitors, yet no tool provides deterministic capture-replay + cheap non-LLM assertions + hard CI/CD blocking as a first-class product. Evalcraft fills this gap by merging VCR.py's record/replay paradigm, pytest's developer experience, and Braintrust/DeepEval's eval scoring into a single, framework-agnostic system.

    Product vision: Every team shipping AI agents tests them with the same rigor they test REST APIs today — automatically, cheaply, on every commit.

    2. User Personas¶

    Persona 1: Solo AI Developer ("Sam")¶

    Profile: Senior engineer building a side-project or early-stage agent. Uses LangGraph or OpenAI Agents SDK. Working alone or with 1-2 collaborators.

    Pain points: - Tests agents by running them manually and eyeballing output - Spends $5-20/day on LLM API calls during development iteration - No way to regression-test after prompt changes - Afraid to refactor because "it was working before"

    Jobs to be done: - Record a working agent session once, replay it free forever - Get a pass/fail signal on agent behavior without re-calling the LLM - Run tests locally in <5 seconds

    Success metric: Time-to-first-test < 5 minutes from pip install

    Persona 2: Startup Engineering Team ("Team Neon", 3-10 devs)¶

    Profile: Series A-B startup with multiple agents in production (customer support, data extraction, code generation). 1-2 dedicated "agent engineers", rest are full-stack.

    Pain points: - CI/CD pipeline exists for code but agents aren't tested - Deployed a prompt change that broke 3 customer workflows — caught by user complaint, not testing - Eval costs ($1-10/CI run on LLM judges) make per-commit testing uneconomical - Different team members have different intuitions about "good agent output"

    Jobs to be done: - Shared golden test sets that define expected agent behavior - CI/CD gate that blocks merges when agent behavior regresses - Cost <$0.01/test run for deterministic assertions; LLM judges only for ambiguous cases - Dashboard showing agent quality trends over time

    Success metric: Agent regression caught in CI before merge; eval cost <$50/month for team

    Persona 3: Enterprise ML Platform Team ("Platform Corp", 50+ devs)¶

    Profile: Large organization with 10+ agents across business units. Dedicated ML platform team provides infrastructure. Compliance requirements (SOC 2, GDPR). Multiple environments (staging, prod).

    Pain points: - No standardized testing framework across agent teams - Compliance requires evidence of pre-deployment validation - Agent failures in production cost $10K-100K per incident - Need audit trail of what was tested, when, by whom

    Jobs to be done: - Standardized eval framework that all agent teams adopt - SSO, RBAC, audit logs for compliance - Self-hosted / VPC deployment option - Integration with existing CI/CD (Jenkins, GitLab CI, GitHub Actions) - Cross-agent benchmarking dashboard

    Success metric: 100% of agent deployments gated by eval suite; SOC 2 audit evidence

    Persona 4: QA Engineer at AI Company ("Quinn")¶

    Profile: QA engineer transitioning from traditional testing to AI agent validation. May not write Python daily. Needs accessible tooling.

    Pain points: - Traditional testing tools don't work for non-deterministic agent output - Doesn't know how to evaluate "good enough" agent responses - Needs to create test cases without deep framework knowledge - Wants visual tools, not just CLI

    Jobs to be done: - Visual dashboard to review agent traces and annotate pass/fail - Test case creation without writing code (YAML/GUI-based) - Human-in-the-loop annotation workflows - Export QA reports for stakeholder review

    Success metric: Can create and run agent test suite without writing Python

    3. Core Feature Specifications¶

    3.1 Capture System¶

    What it does: Records all LLM calls, tool executions, and agent state transitions during a live agent run into a "cassette" file (inspired by VCR.py).

    Data captured per interaction:

    Data Point Format Storage
    LLM requests (messages, model, params) Structured YAML/JSON Cassette file
    LLM responses (content, tool calls, usage) Structured YAML/JSON Cassette file
    Tool inputs and outputs Structured YAML/JSON Cassette file
    Agent state transitions Event stream Cassette file
    Timestamps and latency ISO 8601 + ms Cassette file
    Token usage and cost estimates Numeric Cassette file
    Framework metadata (name, version) String Cassette file header

    Recording modes (inspired by VCR.py):

    Mode Behavior Use Case
    all Always record, overwrite cassette Fresh recording
    once Record if cassette missing, replay if present Default for development
    new_episodes Record new unmatched requests, replay known ones Incremental development
    none Replay-only, fail on unrecorded requests CI/CD (deterministic)

    Privacy & sanitization: - Automatic: API keys, Bearer tokens, common secret patterns stripped before write - PII detection: Email, phone, SSN, credit card patterns → {{REDACTED_EMAIL}} placeholders - Custom rules: Developer-defined regex patterns for domain-specific secrets - ID normalization: Dynamic IDs (UUIDs, DB IDs) → stable placeholders {{NORMALIZED_ID_1}} - Cassettes are human-readable YAML — developers can inspect and edit them - .evalcraftignore file to exclude paths/data from recording (like .gitignore)

    User story (US-CAP-1): As a developer, I want to record my agent's behavior with a single decorator so that I can replay it later without making real API calls.

    Acceptance criteria: - [ ] @evalcraft.capture("path/to/cassette.yaml") records all LLM + tool interactions - [ ] Cassette file is valid YAML readable by humans - [ ] API keys and Bearer tokens are automatically stripped - [ ] Recording completes in <100ms overhead above agent runtime - [ ] Cassette includes full metadata (framework, version, timestamps, cost)

    User story (US-CAP-2): As a developer, I want my cassettes to automatically strip sensitive data so that I can safely commit them to version control.

    Acceptance criteria: - [ ] Default sanitizers catch OpenAI, Anthropic, Google API key patterns - [ ] PII sanitizer detects email, phone, SSN with >95% recall - [ ] Custom sanitization rules via evalcraft.toml config - [ ] Normalized IDs are deterministic (same input → same placeholder) - [ ] evalcraft sanitize --verify <cassette> validates no secrets remain

    3.2 Replay System¶

    What it does: Replays a recorded cassette to re-execute the agent with identical LLM responses and tool outputs, without making any network calls. Enables deterministic regression testing.

    Determinism guarantees: 1. Network firewall: During replay, all outbound HTTP to LLM providers is blocked. Any unrecorded request = immediate test failure with clear error message. 2. Sequential matching: Requests matched to cassette entries in recorded order (VCR.py pattern). 3. Matching strategies: - exact — byte-for-byte request body match - body_exact — exact body, ignore headers/timestamps - body_semantic — embedding-based similarity (threshold 0.95) for tolerance to minor prompt changes - structural — match message structure, ignore content (most flexible) 4. Divergence detection: If agent behavior deviates from recorded trace, the diff is captured and reported.

    Diff algorithm: - Structural diff of agent trace (tool call sequence, LLM message count) - Content diff of LLM outputs (semantic similarity score) - Cost diff (token usage comparison) - Side-by-side diff output in terminal (inspired by git diff)

    User story (US-REP-1): As a developer, I want to replay a recorded agent session so that my tests run in milliseconds without API costs.

    Acceptance criteria: - [ ] @evalcraft.replay("cassette.yaml") replays without network calls - [ ] Replay of a 5-step agent completes in <500ms (vs. 3-5s live) - [ ] API cost during replay is $0.00 - [ ] Unrecorded requests produce clear error: ReplayError: No cassette entry for POST /v1/chat/completions with body hash <hash>. Run with EVALCRAFT_RECORD=1 to re-record.

    User story (US-REP-2): As a developer, I want to see exactly what changed when my agent's behavior diverges from the recording so that I can decide if the change is intentional.

    Acceptance criteria: - [ ] Divergence produces structured diff showing: old vs. new tool calls, LLM outputs, token usage - [ ] Diff is rendered in terminal with colors (green/red like git diff) - [ ] evalcraft diff <cassette1> <cassette2> CLI command for manual comparison - [ ] Diff includes semantic similarity score for LLM output changes

    3.3 Mock System¶

    What it does: Provides configurable fake LLMs and tools for testing agent logic without any external dependencies. Complements capture/replay for scenarios where you want full control over responses.

    Mock types:

    Mock What it replaces Configuration
    MockLLM Any LLM provider Canned responses, conditional rules, response sequences
    MockTool Any tool/function Static responses, input-dependent responses, error simulation
    mock_provider() HTTP-level mock Intercepts all calls to a provider's API

    Limitations (explicitly documented): - Mocks don't capture agent state transitions (use capture/replay for that) - MockLLM response quality is only as good as the developer's mock definitions - Conditional rules use string matching, not semantic understanding - Mock does not simulate token counting or rate limiting by default

    User story (US-MOC-1): As a developer, I want to mock LLM responses so that I can test my agent's branching logic deterministically without an API key.

    Acceptance criteria: - [ ] MockLLM(responses=["a", "b"]) returns responses in sequence - [ ] MockLLM(rules=[...]) matches input patterns to responses - [ ] MockLLM.from_cassette("file.yaml") creates mock from recorded session - [ ] MockTool(name="search", responses={...}) mocks tool by name - [ ] Tests using mocks run without any environment variables or API keys set

    User story (US-MOC-2): As a developer, I want to simulate tool failures so that I can test my agent's error handling paths.

    Acceptance criteria: - [ ] MockTool(name="db_query", error=TimeoutError("connection timed out")) raises on call - [ ] MockTool(name="api", responses={"default": None}, error_rate=0.3) fails 30% of calls - [ ] Agent's error handling behavior is captured in test assertions

    3.4 Eval System¶

    What it does: Scores agent outputs using pluggable evaluators. Supports cheap deterministic scorers (tool call checks, regex, classifiers) and expensive LLM-as-judge scorers, with a tiered pipeline that runs cheap evals first.

    Supported evaluators:

    Scorer Type Cost/eval Latency Use Case
    ExactMatch Deterministic $0.000 <1ms Binary correct/incorrect
    RegexScorer Deterministic $0.000 <1ms Output format validation
    ToolUseCorrectness Deterministic $0.000 <1ms Tool call path assertion
    TrajectoryScorer Deterministic $0.000 <1ms Step count, token bounds
    JSONSchemaValidator Deterministic $0.000 <1ms Structured output validation
    ClassifierScorer ML (local) $0.000 ~5ms Trained quality classifier
    SemanticSimilarity Embedding ~$0.0001 ~50ms Output similarity to reference
    LLMJudge LLM-as-judge ~$0.005 ~2s Open-ended quality assessment
    CompositeScorer Aggregated Varies Varies Weighted multi-scorer

    Custom eval API:

    from evalcraft.eval.scorers import Scorer, EvalResult

class MyCustomScorer(Scorer):
    name = "my_scorer"
    threshold = 0.8

    async def score(self, input, output, expected=None, trace=None, context=None):
        # Your custom scoring logic
        passed = "error" not in output.lower()
        return EvalResult(
            score=1.0 if passed else 0.0,
            passed=passed,
            scorer_name=self.name,
            explanation="Output contained error keyword" if not passed else None,
        )

    Tiered evaluation pipeline (recommended): 

    Tier 1: Deterministic ($0.000/eval, <1ms)
  → ToolUseCorrectness, TrajectoryScorer, RegexScorer, JSONSchemaValidator
  → If ALL pass → GREEN (deploy)
  → If ANY fail → RED (block immediately, no need for expensive evals)

Tier 2: ML-local ($0.000/eval, ~5ms)
  → ClassifierScorer, SemanticSimilarity
  → If pass → GREEN
  → If fail → proceed to Tier 3

Tier 3: LLM-as-Judge ($0.005/eval, ~2s)
  → LLMJudge with custom rubric
  → Final pass/fail decision

    User story (US-EVL-1): As a developer, I want to assert that my agent called the right tools in the right order so that I can catch behavioral regressions without using an LLM judge.

    Acceptance criteria: - [ ] assert_tool_called("search", args={"query": "weather"}) validates tool was called with args - [ ] assert_tool_called("search").before("summarize") validates ordering - [ ] assert_no_tool_called("delete_user") validates forbidden tools not invoked - [ ] assert_llm_calls_count(lte=3) validates LLM call budget - [ ] assert_total_tokens(lte=2000) validates token budget - [ ] All deterministic assertions execute in <1ms with $0.00 cost

    User story (US-EVL-2): As an engineering team, I want to run dataset-driven evaluations against a golden set so that I can benchmark agent quality across releases.

    Acceptance criteria: - [ ] evalcraft.evaluate(dataset=..., task=..., scorers=[...]) runs full eval suite - [ ] Dataset supports YAML, JSON, CSV, and inline Python list formats - [ ] Results include per-case scores, aggregate statistics (mean, P50, pass rate) - [ ] Results uploadable to Evalcraft Cloud for historical tracking - [ ] Concurrent evaluation (configurable parallelism) for speed

    User story (US-EVL-3): As a developer, I want to create custom scorers with a simple Python interface so that I can evaluate domain-specific quality criteria.

    Acceptance criteria: - [ ] Custom scorer requires implementing 1 method: async def score(...) -> EvalResult - [ ] Scorers compose via CompositeScorer with weighted aggregation - [ ] Custom scorers work in CLI, pytest, and CI/CD without modification - [ ] Example scorers provided for: sentiment, toxicity, factual accuracy, format compliance

    3.5 CI/CD Integration¶

    What it does: Provides GitHub Actions (primary), GitLab CI, and generic CI integration that runs agent tests on every push/PR and gates deployment on pass/fail.

    GitHub Actions integration spec:

    # .github/workflows/evalcraft.yml
name: Agent Tests
on: [push, pull_request]

jobs:
  agent-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install evalcraft[pytest]
      - name: Run agent tests
        run: evalcraft run --ci
        env:
          EVALCRAFT_API_KEY: ${{ secrets.EVALCRAFT_API_KEY }}  # Optional: cloud upload
      - name: Upload results
        if: always()
        uses: evalcraft/action@v1
        with:
          results-path: .evalcraft/results/
          comment-on-pr: true     # Post score card as PR comment
          fail-on-regression: true # Block merge if scores dropped

    PR comment format (score card):

    ## Evalcraft Results — 42 tests, 41 passed, 1 failed

| Suite | Tests | Pass | Fail | Avg Score | Δ vs main |
|-------|-------|------|------|-----------|-----------|
| customer_support | 15 | 15 | 0 | 0.94 | +0.02 |
| data_extraction | 12 | 12 | 0 | 0.91 | — |
| code_generation | 15 | 14 | 1 | 0.87 | -0.05 ⚠️ |

❌ **1 regression detected** in `code_generation`:
  - `test_python_function_generation`: ToolUseCorrectness score 0.0 → expected `write_file` tool call, got `read_file` instead

🔗 [View full results on Evalcraft Cloud](https://app.evalcraft.dev/runs/abc123)

    Blocking rules: 1. Hard block: Any test with @pytest.mark.evalcraft(blocking=True) that fails = merge blocked 2. Regression block: If any scorer's aggregate score drops by >X% vs. main branch = merge blocked (configurable threshold) 3. Budget block: If total eval cost exceeds configured budget per run = warning (not block by default) 4. Custom rules: YAML-configurable in evalcraft.toml:

    [ci]
block_on_failure = true
regression_threshold = 0.05  # Block if any score drops >5%
max_cost_per_run = 1.00      # Warn if eval cost exceeds $1
comment_on_pr = true
upload_results = true

    User story (US-CI-1): As an engineering team, I want agent tests to run on every pull request and block merges when quality regresses so that we never ship broken agent behavior.

    Acceptance criteria: - [ ] evalcraft/action@v1 GitHub Action auto-discovers and runs agent tests - [ ] PR comment posted with score card showing per-suite results - [ ] Merge blocked if any blocking=True test fails - [ ] Merge blocked if aggregate score drops >5% vs. main (configurable) - [ ] Total CI time for 50 replay-mode tests < 30 seconds - [ ] Total CI cost for 50 replay-mode tests < $0.01

    User story (US-CI-2): As a developer, I want zero-config CI setup so that I can add agent testing to my pipeline in under 5 minutes.

    Acceptance criteria: - [ ] evalcraft init --ci github generates .github/workflows/evalcraft.yml - [ ] Default config works without Evalcraft Cloud account (local results only) - [ ] First CI run succeeds with zero configuration beyond pip install evalcraft - [ ] Clear error messages if cassettes are missing: "No cassette found. Run evalcraft record locally first."

    3.7 Golden Sets¶

    What it does: Versioned collections of test cases (input/expected output pairs) that define correct agent behavior. Golden sets are the "ground truth" for agent evaluation.

    Format:

    # golden_sets/customer_support.yaml
name: "Customer Support Core"
version: 3
description: "Core test cases for customer support agent"
author: "sam@company.com"
created_at: "2026-02-20"
updated_at: "2026-03-05"

cases:
  - id: "cs_001"
    input: "I want to cancel my subscription"
    expected:
      output_contains: ["cancelled", "subscription"]
      tools_called: ["lookup_customer", "cancel_subscription"]
      tools_not_called: ["charge_customer"]
      max_steps: 6
      max_tokens: 3000
    tags: ["cancel", "happy-path"]
    priority: "critical"

  - id: "cs_002"
    input: "I was charged twice this month"
    expected:
      output_contains: ["refund", "apologize"]
      tools_called: ["lookup_customer", "check_billing_history"]
      tools_called_in_order: true
      max_steps: 8
    tags: ["billing", "error-case"]
    priority: "high"

    Versioning: - Golden sets stored as YAML files in the repo (git-versioned) - Cloud dashboard provides visual editor + version history - evalcraft golden diff v2 v3 shows what changed between versions - Golden sets can be shared across projects via Evalcraft Cloud

    Collaboration: - Team members can propose golden set changes via PR (golden set file is code-reviewed) - Dashboard provides "suggest test case" from production traces - Export from production: observed good agent runs → golden set candidate

    User story (US-GLD-1): As a team, I want to maintain a versioned set of expected agent behaviors so that all team members agree on what "correct" means.

    Acceptance criteria: - [ ] Golden sets stored as YAML in repo, versioned by git - [ ] Cloud dashboard provides visual editor for non-developers - [ ] Version history shows who changed what and when - [ ] evalcraft golden validate <file> checks golden set schema - [ ] Golden sets support tags for filtering (evalcraft run --tags critical)

    4.2 SDK API Surface¶

    Python SDK (evalcraft)¶

    # ── Core API ──
import evalcraft

# Decorators
@evalcraft.capture(cassette, mode="once", sanitize=True, normalize_ids=True)
@evalcraft.replay(cassette, match_on="body_semantic", strict=False)
@evalcraft.mock(llm=MockLLM(...), tools={...})

# Context managers
with evalcraft.recording(cassette, **options) as session: ...
with evalcraft.replaying(cassette, **options) as session: ...

# Eval
results = await evalcraft.evaluate(name, dataset, task, scorers, **options)

# Configuration
evalcraft.configure(cassette_dir, default_mode, sanitize_keys, **options)

# Session assertions
session.assert_tool_called(name, args=None, count=None)
session.assert_tool_called(name).before(other_name)
session.assert_no_tool_called(name)
session.assert_llm_calls_count(eq=None, lte=None, gte=None)
session.assert_total_tokens(lte=None)
session.assert_output_contains(text)
session.assert_output_matches(regex)
session.assert_output_schema(json_schema)

# Pytest plugin (auto-registered)
# Markers: @pytest.mark.evalcraft(mode="auto", blocking=True)
# Fixtures: evalcraft_session, evalcraft_cassette, mock_llm

# Framework adapters
from evalcraft.adapters import LangGraphAdapter, OpenAIAgentsAdapter, ClaudeAgentAdapter

format = "yaml"                   # yaml | json

[replay]
match_strategy = "body_semantic"  # exact | body_exact | body_semantic | structural
similarity_threshold = 0.95
strict = false                    # If true, fail on any unmatched request

[eval]
default_scorers = ["ToolUseCorrectness", "TrajectoryScorer"]
concurrency = 5
timeout_per_case = 30

[ci]
block_on_failure = true
regression_threshold = 0.05       # Block if score drops >5%
max_cost_per_run = 1.00
comment_on_pr = true
upload_results = true

[cloud]
api_url = "https://api.evalcraft.dev"
# api_key set via EVALCRAFT_API_KEY env var (never in config file)

evalcraft run -v
